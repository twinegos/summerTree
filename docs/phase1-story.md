# Phase 1 유저 스토리 — 데이터 모델 & 관리자 페이지

**프로젝트**: summerTree 식물 가게 모바일 웹앱
**Phase 범위**: DB 스키마 설계, 관리자 인증, 식물/카테고리 CRUD
**기술 스택**: Nuxt 4 + TypeScript + Tailwind CSS + Supabase
**작성일**: 2026-06-03

---

## 등장인물 (Personas)

| 역할 | 설명 |
|------|------|
| **관리자** | 식물 가게 운영자. Supabase Auth로 로그인하여 식물 정보와 재고를 관리한다 |
| **고객** | Phase 1 범위 외 — 식물 상세 페이지/장바구니/결제는 Phase 2~4에서 다룸 |

---

## DB 스키마 전제

Phase 1 구현에 필요한 테이블 목록 (스펙 단계에서 상세 정의):

- `categories` — 식물 카테고리 (다육식물, 관엽식물 등)
- `plants` — 식물 상품 정보 (이름, 설명, 가격, 재고, 이미지 등)
- `orders` — 주문 (Phase 4에서 사용하지만 스키마는 Phase 1에서 생성)
- `order_items` — 주문 상품 상세
- `cart_items` — 장바구니 (Phase 3에서 사용하지만 스키마는 Phase 1에서 생성)

## Story 1 — 관리자 로그인

**스토리**
나는 **식물 가게 관리자**로서 이메일과 비밀번호로 안전하게 로그인하고 싶다. 왜냐하면 식물 정보와 재고는 인가된 사람만 수정할 수 있어야 하기 때문이다.

**인수 조건**

```
Scenario 1: 정상 로그인
  Given 관리자가 /admin/login 페이지에 접속했다
  When  등록된 이메일과 올바른 비밀번호를 입력하고 로그인 버튼을 누른다
  Then  /admin 대시보드로 이동한다
  And   세션이 유지되어 새로고침 후에도 로그인 상태가 유지된다

Scenario 2: 잘못된 비밀번호
  Given 관리자가 /admin/login 페이지에 있다
  When  이메일은 맞지만 비밀번호가 틀린 경우 로그인을 시도한다
  Then  "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지가 표시된다
  And   페이지를 벗어나지 않는다

Scenario 3: 인증 없이 관리자 페이지 접근
  Given 로그인하지 않은 사용자가
  When  /admin 하위 URL에 직접 접근을 시도한다
  Then  /admin/login 으로 리디렉션된다

Scenario 4: 로그아웃
  Given 관리자가 로그인 상태이다
  When  로그아웃 버튼을 누른다
  Then  세션이 종료되고 /admin/login 으로 이동한다
  And   뒤로가기 버튼을 눌러도 관리자 페이지에 접근되지 않는다
```

**엣지 케이스**
- 이메일 형식이 아닌 문자열 입력 시 클라이언트 유효성 검사로 제한
- 네트워크 오류 시 "연결에 문제가 발생했습니다. 다시 시도해주세요" 표시
- 로그인 처리 중 버튼 비활성화(로딩 상태) — 중복 요청 방지

**범위 외**
- 비밀번호 찾기 / 이메일 재설정 기능
- 2FA(이중 인증)
- 관리자 계정 신규 생성 UI (Supabase 대시보드에서 직접 등록)
- 일반 고객 로그인 (고객은 Phase 1에서 비회원)

## Story 2 — 카테고리 등록

**스토리**
나는 **관리자**로서 식물 카테고리(예: 다육식물, 관엽식물, 선인장)를 등록하고 싶다. 왜냐하면 카테고리를 먼저 만들어야 식물을 분류하여 등록할 수 있기 때문이다.

**인수 조건**

```
Scenario 1: 카테고리 정상 등록
  Given 관리자가 카테고리 관리 페이지에 있다
  When  카테고리 이름(예: "다육식물")과 설명을 입력하고 저장 버튼을 누른다
  Then  새 카테고리가 목록에 추가된다
  And   "카테고리가 등록되었습니다" 알림이 표시된다

Scenario 2: 중복 카테고리 이름
  Given "다육식물" 카테고리가 이미 존재한다
  When  동일한 이름 "다육식물"로 등록을 시도한다
  Then  "이미 존재하는 카테고리 이름입니다" 오류가 표시된다

Scenario 3: 카테고리 이름 없이 저장
  Given 관리자가 카테고리 등록 폼에 있다
  When  이름을 입력하지 않고 저장 버튼을 누른다
  Then  "카테고리 이름은 필수입니다" 유효성 오류가 표시된다
```

**엣지 케이스**
- 카테고리 이름 최대 길이: 50자
- 앞뒤 공백 자동 trim 처리 ("  다육식물  " → "다육식물")

**범위 외**
- 카테고리 순서 변경 (드래그 앤 드롭)
- 카테고리 이미지/아이콘 업로드
- 카테고리 계층 구조 (부모/자식)

## Story 3 — 카테고리 수정

**스토리**
나는 **관리자**로서 기존 카테고리의 이름이나 설명을 수정하고 싶다. 왜냐하면 오타를 고치거나 더 명확한 이름으로 변경해야 할 수 있기 때문이다.

**인수 조건**

```
Scenario 1: 카테고리 이름 수정
  Given 관리자가 카테고리 목록에서 수정할 카테고리를 선택했다
  When  이름을 변경하고 저장한다
  Then  변경된 이름이 목록에 즉시 반영된다
  And   해당 카테고리에 속한 식물의 카테고리 표시도 자동으로 업데이트된다

Scenario 2: 수정 취소
  Given 관리자가 카테고리 수정 폼을 열었다
  When  내용을 변경하다가 취소 버튼을 누른다
  Then  변경 사항이 적용되지 않고 이전 값이 유지된다
```

**엣지 케이스**
- 수정 중 다른 관리자가 동일 항목을 삭제한 경우: "이미 삭제된 카테고리입니다" 오류 표시 (Phase 1에서는 단일 관리자이므로 낮은 우선순위)

**범위 외**
- 수정 이력(audit log) 기록

## Story 4 — 카테고리 삭제

**스토리**
나는 **관리자**로서 사용하지 않는 카테고리를 삭제하고 싶다. 왜냐하면 더 이상 취급하지 않는 식물 분류를 정리해야 하기 때문이다.

**인수 조건**

```
Scenario 1: 빈 카테고리 삭제
  Given 해당 카테고리에 등록된 식물이 없다
  When  관리자가 삭제 버튼을 누르고 확인 다이얼로그에서 확인을 선택한다
  Then  카테고리가 목록에서 제거된다

Scenario 2: 식물이 있는 카테고리 삭제 시도
  Given 해당 카테고리에 식물이 1개 이상 등록되어 있다
  When  관리자가 삭제를 시도한다
  Then  "이 카테고리에 식물 N개가 등록되어 있습니다. 먼저 식물을 다른 카테고리로 이동하거나 삭제해주세요" 오류가 표시된다
  And   카테고리는 삭제되지 않는다

Scenario 3: 삭제 취소
  Given 관리자가 삭제 확인 다이얼로그를 열었다
  When  취소를 선택한다
  Then  카테고리가 삭제되지 않는다
```

**엣지 케이스**
- 삭제 직전 식물이 새로 추가된 경우: DB 외래 키 제약으로 서버에서 차단, 오류 메시지 표시

**범위 외**
- 카테고리 소프트 삭제(is_deleted 플래그) — Phase 1에서는 하드 삭제

## Story 5 — 식물 등록

**스토리**
나는 **관리자**로서 새 식물 상품을 등록하고 싶다. 왜냐하면 QR 코드 생성의 전제가 되는 식물 정보(이름, 설명, 가격 등)를 시스템에 저장해야 하기 때문이다.

**인수 조건**

```
Scenario 1: 필수 정보 입력 후 정상 등록
  Given 관리자가 식물 등록 페이지에 있다
  When  아래 필수 항목을 모두 입력하고 등록 버튼을 누른다
        - 이름 (필수)
        - 카테고리 (필수, 드롭다운 선택)
        - 가격 (필수, 0 이상의 정수)
        - 재고 (필수, 0 이상의 정수)
  Then  새 식물이 DB에 저장된다
  And   식물 목록 페이지로 이동하며 "식물이 등록되었습니다" 알림이 표시된다

Scenario 2: 선택 항목 포함 등록
  Given 관리자가 식물 등록 페이지에 있다
  When  필수 항목에 더해 아래 선택 항목도 입력한다
        - 간단 설명 (텍스트)
        - 상세 설명 (마크다운 또는 긴 텍스트)
        - 키우는 방법 (텍스트)
        - 주의사항 (텍스트)
        - 이미지 1장 이상 업로드
  Then  모든 정보가 저장되고 식물 상세 페이지에서 확인 가능하다

Scenario 3: 이미지 업로드
  Given 관리자가 이미지 업로드 영역에 이미지를 선택했다
  When  jpg, png, webp 형식의 이미지 파일을 선택한다 (최대 5MB)
  Then  Supabase Storage에 업로드되고 미리보기가 표시된다
  And   식물 등록 완료 시 이미지 URL이 DB에 함께 저장된다

Scenario 4: 필수 항목 누락
  Given 관리자가 식물 등록 폼에 있다
  When  이름을 입력하지 않고 등록을 시도한다
  Then  "식물 이름은 필수입니다" 유효성 오류가 표시된다
  And   등록이 진행되지 않는다

Scenario 5: 가격에 음수 입력
  Given 관리자가 가격 필드에 음수를 입력한다
  When  등록을 시도한다
  Then  "가격은 0 이상이어야 합니다" 오류가 표시된다

Scenario 6: 카테고리 미선택
  Given 관리자가 카테고리를 선택하지 않았다
  When  등록을 시도한다
  Then  "카테고리를 선택해주세요" 오류가 표시된다
```

**엣지 케이스**
- 이미지 용량 초과(5MB 이상): "이미지 크기는 5MB 이하여야 합니다" 오류, 업로드 차단
- 지원하지 않는 이미지 형식(gif, bmp 등): "jpg, png, webp 형식만 지원합니다" 오류
- 가격 필드에 문자 입력: 숫자만 허용하도록 input type="number" 처리
- 식물 이름 최대 길이: 100자
- 설명 최대 길이: 2000자
- 이미지 최대 수량: 5장

**범위 외**
- 이미지 다중 순서 변경
- 이미지 크롭/편집
- 식물 복제(기존 식물 기반으로 새 식물 등록)
- QR 코드 생성 (Phase 2에서 구현)
- 재고 이력 추적

## Story 6 — 식물 수정

**스토리**
나는 **관리자**로서 등록된 식물의 정보를 수정하고 싶다. 왜냐하면 가격 변경, 재고 업데이트, 설명 보완이 필요할 수 있기 때문이다.

**인수 조건**

```
Scenario 1: 식물 정보 수정
  Given 관리자가 식물 목록에서 특정 식물의 수정 버튼을 눌렀다
  When  원하는 필드(예: 가격, 재고)를 변경하고 저장 버튼을 누른다
  Then  변경된 정보가 DB에 반영된다
  And   "수정이 완료되었습니다" 알림이 표시된다

Scenario 2: 이미지 교체
  Given 관리자가 식물 수정 폼에서 기존 이미지를 확인한다
  When  기존 이미지를 삭제하고 새 이미지를 업로드한다
  Then  Supabase Storage에서 기존 이미지가 삭제되고 새 이미지가 저장된다

Scenario 3: 이미지 추가
  Given 식물에 이미지가 이미 있다
  When  추가 이미지를 업로드한다 (최대 5장 제한)
  Then  새 이미지가 기존 이미지 목록에 추가된다

Scenario 4: 수정 취소
  Given 관리자가 여러 필드를 변경했다
  When  취소 버튼을 누른다
  Then  모든 변경 사항이 원래 값으로 복원되고 목록 페이지로 이동한다
```

**엣지 케이스**
- 이미지 5장 초과 추가 시도: "이미지는 최대 5장까지 등록 가능합니다" 오류
- 재고를 0으로 수정 시: 정상 저장 (품절 상태로 처리)
- 저장 중 네트워크 오류: "저장에 실패했습니다. 다시 시도해주세요" 표시, 입력 내용 유지

**범위 외**
- 수정 이력(revision history) 기록
- 여러 식물 일괄 수정

## Story 7 — 식물 삭제

**스토리**
나는 **관리자**로서 더 이상 판매하지 않는 식물을 삭제하고 싶다. 왜냐하면 취급하지 않는 상품을 목록에서 제거해야 하기 때문이다.

**인수 조건**

```
Scenario 1: 정상 삭제
  Given 관리자가 식물 목록에서 삭제할 식물을 선택했다
  When  삭제 버튼을 누르고 확인 다이얼로그에서 "삭제"를 선택한다
  Then  식물이 DB에서 삭제된다
  And   Supabase Storage에 저장된 해당 식물의 이미지도 삭제된다
  And   식물 목록에서 제거된다

Scenario 2: 삭제 취소
  Given 관리자가 삭제 확인 다이얼로그를 열었다
  When  "취소"를 선택한다
  Then  식물이 삭제되지 않는다

Scenario 3: 주문 내역이 있는 식물 삭제
  Given 해당 식물이 과거 주문 내역에 포함되어 있다
  When  관리자가 삭제를 시도한다
  Then  주문 내역은 보존되고 식물 정보만 "삭제된 상품"으로 표시된다
  (소프트 삭제: is_deleted = true, DB에서 물리적으로 제거하지 않음)
```

**엣지 케이스**
- 이미 삭제된 식물에 대한 요청(새로고침 후 재시도 등): "이미 삭제된 식물입니다" 오류
- 이미지 삭제 실패 시: 식물 DB 삭제는 성공하되, 이미지 삭제 오류를 로그로 기록 (사용자에게는 성공으로 표시, 이미지는 주기적 정리 작업으로 처리)

**범위 외**
- 삭제된 식물 복구(휴지통) 기능
- 여러 식물 일괄 삭제

## Story 8 — 식물 목록 조회 (관리자)

**스토리**
나는 **관리자**로서 등록된 모든 식물의 목록을 한눈에 보고 싶다. 왜냐하면 재고 현황을 파악하고 수정/삭제할 식물을 빠르게 찾아야 하기 때문이다.

**인수 조건**

```
Scenario 1: 목록 조회
  Given 관리자가 /admin/plants 페이지에 접속한다
  When  페이지가 로드된다
  Then  등록된 식물 목록이 표시된다
  And   각 식물 항목에 이름, 카테고리, 가격, 재고, 대표 이미지 썸네일이 보인다

Scenario 2: 카테고리 필터
  Given 식물 목록이 표시되어 있다
  When  특정 카테고리를 필터로 선택한다
  Then  해당 카테고리 식물만 표시된다

Scenario 3: 식물 검색
  Given 식물 목록이 표시되어 있다
  When  검색창에 식물 이름의 일부를 입력한다
  Then  이름에 해당 키워드가 포함된 식물만 표시된다

Scenario 4: 등록된 식물 없음
  Given 아직 식물이 하나도 등록되지 않았다
  When  관리자가 목록 페이지에 접속한다
  Then  "등록된 식물이 없습니다. 첫 번째 식물을 등록해보세요!" 메시지와 등록 버튼이 표시된다
```

**엣지 케이스**

- 재고 0인 식물: 목록에서 "품절" 배지로 표시

## 기술 노트 (스펙 작성 시 참고)

- **인증**: Supabase Auth의 `signInWithPassword` 사용. 세션은 `useSupabaseUser()` 컴포저블로 관리
- **라우트 보호**: Nuxt 미들웨어 (`middleware/auth.ts`) 에서 `/admin/**` 경로 인증 확인
- **이미지 저장**: Supabase Storage `plant-images` 버킷. 파일명은 `{plant_id}/{uuid}.{ext}` 형식
- **Storage 삭제**: 식물 삭제 시 서버 API에서 Storage 파일 삭제 후 DB 레코드 삭제 순서로 처리
- **소프트 삭제**: 주문 내역이 있는 식물은 `is_deleted = true` 처리. 주문 내역 없는 식물은 하드 삭제
